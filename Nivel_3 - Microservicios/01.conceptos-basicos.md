# 🏗️ Guía Esencial de Microservicios

Los microservicios son un enfoque arquitectónico donde una aplicación se divide en servicios pequeños, independientes y especializados que se comunican entre sí.

---

## 1. Fundamentos de Comunicación

### **API (Application Programming Interface)**
Es el contrato o menú que define cómo se puede interactuar con un servicio. Permite que un microservicio exponga sus funciones al mundo.

### **Protocolo HTTP**
Es el idioma estándar de la web. Los microservicios lo usan para enviarse mensajes.

### **Verbos HTTP (Métodos)**
Indican la acción que se quiere realizar sobre un recurso:
* **GET:** Consultar o leer información (ej. ver un catálogo).
* **POST:** Crear algo nuevo (ej. registrar un pedido).
* **PUT / PATCH:** Actualizar algo existente (ej. cambiar una dirección).
* **DELETE:** Eliminar un recurso (ej. borrar un perfil).

### **JSON (JavaScript Object Notation)**
Es el formato de datos más usado para el intercambio de información. Es ligero y fácil de leer para humanos y máquinas.

---

## 2. Infraestructura y Orquestación

### **API Gateway (El Recepcionista)**
Punto de entrada único para todas las peticiones externas. Se encarga de la seguridad, autenticación y de dirigir el tráfico al servicio correcto.

### **Service Discovery (El Directorio)**
Como las direcciones IP de los servicios cambian constantemente, este sistema mantiene una lista actualizada de dónde está cada microservicio para que puedan encontrarse entre sí.

### **Message Broker (El Buzón)**
Permite la comunicación **asíncrona**. Un servicio deja un mensaje en una "cola" (como RabbitMQ o Kafka) y otro lo recoge cuando está listo, evitando que el sistema se bloquee esperando respuestas.

### **Contenedores y Docker**
Empaquetan el código y sus dependencias en una unidad estándar que funciona igual en cualquier computadora o servidor.

---

## 3. Principios de Diseño (Las "Reglas")

### **Base de Datos por Servicio**
Cada microservicio debe ser dueño de sus propios datos. Esto evita que, si una base de datos falla, se caiga todo el sistema.

### **Escalabilidad Horizontal**
En lugar de comprar un servidor más grande (Vertical), los microservicios permiten crear muchas copias pequeñas del mismo servicio para atender a más usuarios.



### **Stateless (Sin Estado)**
Los servicios no deben guardar información de la sesión en su memoria interna. Cada petición debe ser independiente para que cualquier "copia" del servicio pueda atenderla.

### **Circuit Breaker (El Fusible)**
Si un servicio empieza a fallar, el Circuit Breaker corta las peticiones hacia él para evitar que el fallo se propague y bloquee otros servicios saludables.

---

## 📊 Resumen de Conceptos Clave

| Concepto | Función Principal | Analogía |
| :--- | :--- | :--- |
| **API Gateway** | Punto de entrada único | Recepcionista de un edificio |
| **Service Discovery** | Localizar servicios | Directorio telefónico / GPS |
| **Message Broker** | Comunicación diferida | Buzón de correos / Mensajería |
| **Load Balancer** | Repartir el trabajo | Jefe de fila en un banco |
| **Circuit Breaker** | Evitar colapsos | Fusible eléctrico |

---

## 🔢 Códigos de Estado (Status Codes)
Para saber qué pasó con una petición, se usan estos códigos:
* **200 (OK):** Todo salió bien.
* **201 (Created):** Se creó un recurso con éxito.
* **400 (Bad Request):** Enviaste algo mal en la petición.
* **404 (Not Found):** Lo que buscas no existe.
* **500 (Internal Error):** El microservicio tuvo un fallo interno.
