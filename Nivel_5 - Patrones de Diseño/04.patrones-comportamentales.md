# Patrones de Diseño de Comportamiento en Java

Los patrones de comportamiento se encargan de la interacción y comunicación entre objetos, así como de la asignación de responsabilidades. Se centran en cómo los objetos colaboran para llevar a cabo tareas complejas.

---

## Patrón Chain of Responsibility
**Propósito:** Permite pasar solicitudes a lo largo de una cadena de manejadores. Al recibir una solicitud, cada manejador decide si la procesa o la pasa al siguiente manejador de la cadena.
- **Ventajas:** Reduce el acoplamiento; permite añadir o cambiar manejadores dinámicamente.
- **Desventajas:** No se garantiza que la solicitud sea procesada si llega al final de la cadena sin éxito.
- **Casos de uso:** Sistemas de filtrado de logs, validación de formularios secuencial, middleware HTTP.

```java
abstract class Handler {
    protected Handler next;
    public void setNext(Handler next) { this.next = next; }
    public abstract void handle(String request);
}

class AuthHandler extends Handler {
    public void handle(String request) {
        if (request.equals("AUTH")) System.out.println("Autenticación aprobada");
        else if (next != null) next.handle(request);
    }
}
```

## Patrón Command
**Propósito:** Encapsula una solicitud como un objeto, permitiendo parametrizar clientes con diferentes solicitudes, encolarlas o registrar operaciones.
- **Ventajas:** Desacopla el objeto que invoca la operación del que sabe cómo realizarla; facilita la implementación del "Deshacer" (Undo).
- **Desventajas:** El código puede volverse más complejo debido a la creación de múltiples clases para cada comando.
- **Casos de uso:** Implementación de botones de interfaz, operaciones transaccionales, macros.

```java
interface Command { void execute(); }

class Light { public void on() { System.out.println("Luz encendida"); } }

class LightOnCommand implements Command {
    private Light light;
    public LightOnCommand(Light light) { this.light = light; }
    public void execute() { light.on(); }
}
```

## Patrón Iterator
**Propósito:** Permite recorrer elementos de una colección sin exponer su estructura interna.
- **Ventajas:** Proporciona una interfaz uniforme para recorrer diferentes estructuras de datos.
- **Desventajas:** Puede resultar excesivo para colecciones simples que ya tienen métodos de recorrido eficientes.
- **Casos de uso:** Recorrido de listas, árboles o grafos en librerías de colecciones.

```java
import java.util.Iterator;
import java.util.List;

class NameRepository implements Iterable<String> {
    private List<String> names = List.of("Juan", "Maria", "Pedro");
    public Iterator<String> iterator() { return names.iterator(); }
}
```

## Patrón Mediator
**Propósito:** Restringe las comunicaciones directas entre objetos y los obliga a colaborar solo a través de un objeto mediador.
- **Ventajas:** Centraliza el control de la comunicación; reduce las dependencias entre componentes.
- **Desventajas:** El mediador puede convertirse en un "Objeto Dios" excesivamente complejo y difícil de mantener.
- **Casos de uso:** Torres de control de tráfico aéreo, diálogos de interfaz de usuario complejos.

```java
interface ChatMediator { void sendMessage(String msg, User user); }

abstract class User {
    protected ChatMediator mediator;
    public User(ChatMediator med) { this.mediator = med; }
    public abstract void send(String msg);
}
```

## Patrón Memento
**Propósito:** Permite capturar y guardar el estado interno de un objeto sin revelar su implementación, para poder restaurarlo más tarde.
- **Ventajas:** Facilita la creación de puntos de restauración (Checkpoints) de forma segura.
- **Desventajas:** El consumo de memoria puede ser alto si se guardan muchos estados de objetos grandes.
- **Casos de uso:** Editores de texto (Ctrl+Z), estados de juegos, recuperación ante errores.

```java
class EditorMemento {
    private final String content;
    public EditorMemento(String content) { this.content = content; }
    public String getContent() { return content; }
}
```

## Patrón Observer
**Propósito:** Define un mecanismo de suscripción para notificar a varios objetos sobre cualquier evento que le suceda al objeto que están observando.
- **Ventajas:** Cumple con el principio de Open/Closed; establece relaciones dinámicas entre objetos.
- **Desventajas:** Los observadores son notificados en un orden aleatorio y pueden recibir actualizaciones irrelevantes.
- **Casos de uso:** Sistemas de noticias, actualizaciones de precios en bolsa, listeners de eventos UI.

```java
import java.util.ArrayList;
import java.util.List;

class NewsAgency {
    private List<Observer> observers = new ArrayList<>();
    public void addObserver(Observer o) { observers.add(o); }
    public void notifyObservers(String news) {
        for (Observer o : observers) o.update(news);
    }
}
interface Observer { void update(String news); }
```

## Patrón State
**Propósito:** Permite que un objeto altere su comportamiento cuando su estado interno cambia. El objeto parecerá haber cambiado de clase.
- **Ventajas:** Elimina condicionales (if/switch) masivos; organiza el código relativo a estados específicos.
- **Desventajas:** Puede ser excesivo si el objeto tiene pocos estados o rara vez cambian.
- **Casos de uso:** Cajeros automáticos, máquinas expendedoras, flujos de pedidos.

```java
interface State { void doAction(); }

class PublishedState implements State {
    public void doAction() { System.out.println("El documento ya está publicado."); }
}
```

## Patrón Strategy
**Propósito:** Define una familia de algoritmos, los encapsula y los hace intercambiables. Permite que el algoritmo varíe independientemente de los clientes que lo utilizan.
- **Ventajas:** Permite cambiar el comportamiento en tiempo de ejecución; evita la herencia múltiple.
- **Desventajas:** El cliente debe conocer las diferencias entre las estrategias para elegir la correcta.
- **Casos de uso:** Diferentes métodos de pago, algoritmos de ordenamiento, filtros de imágenes.

```java
interface PaymentStrategy { void pay(int amount); }

class CreditCardStrategy implements PaymentStrategy {
    public void pay(int amount) { System.out.println("Pagado con Tarjeta: " + amount); }
}

class ShoppingCart {
    public void processOrder(PaymentStrategy strategy, int amount) {
        strategy.pay(amount);
    }
}
```

## Patrón Template Method
**Propósito:** Define el esqueleto de un algoritmo en una operación, delegando algunos pasos a las subclases sin cambiar la estructura del algoritmo.
- **Ventajas:** Fomenta la reutilización de código; centraliza partes comunes del algoritmo.
- **Desventajas:** Puede limitar la flexibilidad de las subclases si el esqueleto es demasiado rígido.
- **Casos de uso:** Generación de reportes en diferentes formatos, frameworks de pruebas unitarias.

```java
abstract class Game {
    abstract void initialize();
    abstract void startPlay();
    abstract void endPlay();

    public final void play() {
        initialize();
        startPlay();
        endPlay();
    }
}
```

## Patrón Visitor
**Propósito:** Permite añadir nuevas operaciones a una estructura de objetos sin modificarla.
- **Ventajas:** Mantiene las operaciones relacionadas juntas; ayuda a cumplir el principio de responsabilidad única.
- **Desventajas:** Es difícil de implementar si la estructura de clases de los objetos cambia con frecuencia.
- **Casos de uso:** Análisis sintáctico (compiladores), exportación de datos de estructuras complejas.

```java
interface Element { void accept(Visitor v); }

interface Visitor { void visit(Element e); }
```
